## Общая статистика
- **Состояние**: ~80+ полей в `state`, >50 мутаций, >70 экшенов, >40 геттеров.
- **Кэш-уровни**: localStorage (справочники), IndexedDB (крупные выборки), in-memory (runtime).
- **Ключевые зависимости**: `CacheInvalidator`, `retryWithExponentialBackoff`, контроллеры API (обычный и simple режим), DTO-конвертеры.
- **Персистенция**: 3 экземпляра `createPersistedState` с разными `paths`, `filter` и `parser`.
- **Синхронизация вкладок**: listener `storage` + `eventBus` события (`company-updated`, `cache:invalidate`).

## Структура и ответственность модулей
- **State**: пользователь, права, уведомления, флаги загрузки, справочники (глобальные и company-scoped), фильтры, меню, настройки канбана, флаги синхронизации.
- **Mutations**: установки коллекций, флагов загрузки, логов, текущей компании, меню, фильтров; сброс company-данных и частичных кэшей.
- **Actions (ядро)**:
  - Инициализация приложения (`initializeApp`): аутентификация, загрузка пользователя, прав, меню, валют и единиц, компаний и текущей компании.
  - Управление компанией (`loadCurrentCompany`, `setCurrentCompany`, `loadCompanyData`): подтягивание справочников, очистка кэшей при смене компании.
  - Инвалидация кэша (`invalidateCache`, `onDataCreate/Update/Delete`, `onUserChange`, `onCompanyChange`).
  - Загрузка справочников (global, company-scoped, products search) с дедупликацией через `loadingFlags` и `waitForLoading`.
- **Getters**: доступ к сущностям, вычисляемые фильтры, округление по компании, фильтрованные пользователи, меню с проверкой прав.

## Загрузка и кэширование данных
- **Глобальные справочники** (`units`, `currencies`, `users`, статусы и т.п.): localStorage, TTL-проверки в `parser` и `reducer`.
- **Company-scoped** (`warehouses`, `cashRegisters`, `clients`, `projects`, категории): инвалидация при смене компании, TTL через `CacheInvalidator` и локальные сбросы.
- **Крупные выборки** (`allProductsData`, `clientsData`, `projectsData`): IndexedDB с `_meta.timestamp`, проверкой TTL и соответствия `projectsDataCompanyId` текущей компании.
- **Слои DTO**: восстановление DTO из plain-кэша при пустом runtime (clients/projects/products search).
- **Загрузка пакетом**: `loadCompanyData` параллелит критичные справочники (склады, кассы) и остальные через `Promise.allSettled`.

---

## Синхронизация и события
- **storage listener**: реагирует на изменение `userSettings/persistedState`, подхватывает `currentCompany` другой вкладки без запросов к API, очищает кэш, перезагружает данные.
- **eventBus**:
  - `company-updated`: перезагрузка списка компаний, обновление текущей компании и версии логотипа.
  - `cache:invalidate`: проксирует в `invalidateCache`.
- **Флаги**: `isSyncingCompanyFromOtherTab` предотвращает гонки; debounce 50 мс для событий storage.

## Проверка соответствия vuex-persistedstate

### Используемые опции

В коде используются следующие опции `createPersistedState`:

#### ✅ Стандартные опции (поддерживаются официально):
- `key` - ключ для хранения в storage
- `storage` - объект хранилища (localStorage, sessionStorage, кастомный)
- `paths` - массив путей для частичного сохранения состояния
- `filter` - функция фильтрации мутаций
- `reducer` - функция преобразования состояния перед сохранением
- `getState` - функция чтения состояния из storage (альтернатива стандартному парсингу)
- `rehydrated` - колбэк после восстановления состояния

#### ⚠️ Потенциально нестандартные опции:
- `assertStorage` - кастомная функция проверки доступности storage
- `fetchBeforeUse` - опция для IndexedDB (может быть нестандартной)

### Проблемы с использованием опций

1. **`assertStorage` не является стандартной опцией vuex-persistedstate**
   - В официальной документации нет упоминания этой опции
   - Используется в 3 местах (строки 1962, 1973, 2109)
   - Может игнорироваться плагином или вызывать ошибки

2. **`fetchBeforeUse` может быть нестандартной опцией**
   - Используется для IndexedDB плагина (строка 2023)
   - Не упоминается в стандартной документации
   - Может не работать как ожидается

3. **Дублирование логики между `reducer` и `paths`**
   - В первом плагине (строки 1846-1967) указаны `paths`, но также используется кастомный `reducer`
   - `reducer` переопределяет логику `paths`, что может быть избыточно

4. **Использование `getState` вместо стандартного парсинга**
   - Во втором плагине (строки 1970-2011) используется `getState` вместо стандартного механизма
   - Это корректно, но усложняет код

5. **IndexedDB storage с async методами**
   - `indexedDBStorage` использует async методы (`getItem`, `setItem`, `removeItem`)
   - Стандартный `vuex-persistedstate` ожидает синхронные методы
   - Может работать некорректно или требовать специальной обработки

### Рекомендации по vuex-persistedstate

1. **Проверить поддержку `assertStorage` и `fetchBeforeUse`**
   - Либо найти документацию/исходники, подтверждающие поддержку
   - Либо удалить эти опции и реализовать проверку иначе

2. **Упростить конфигурацию первого плагина**
   - Убрать дублирование между `paths` и `reducer`
   - Либо использовать только `paths`, либо только `reducer`

3. **Проверить работу с async IndexedDB**
   - Убедиться, что `vuex-persistedstate` корректно работает с async storage
   - Возможно, требуется обертка или другой подход

4. **Добавить обработку ошибок для нестандартных опций**
   - Если опции не поддерживаются, они должны игнорироваться без ошибок
   - Добавить fallback логику

## Выявленные проблемы

### 🔴 Критические
1. **Недостаточная сигнализация о неуспехе загрузок**: `loadCompanyData` ставит `companyData` в `false` в `finally` даже при полном провале; нет явного статуса успеха/ошибки для верхнего уровня.
2. **Стойкость к частичным сбоям прав**: `refreshUserPermissions` пробрасывает исключение, не чистит state; возможна рассинхронизация user/permissions после ошибки сети.
3. **Смена компании без отката частичных изменений**: `setCurrentCompany` при ошибке уже мог очистить кэш и сбросить флаги, но не восстанавливает предыдущие данные.

### 🟡 Средние
4. **Использование нестандартных опций vuex-persistedstate**: `assertStorage` и `fetchBeforeUse` не упоминаются в официальной документации; могут игнорироваться или вызывать проблемы при обновлении плагина.
5. **Дублирование логики между `paths` и `reducer`**: в первом плагине указаны `paths`, но также используется кастомный `reducer`, который переопределяет логику `paths`.
6. **Сложность и дублирование в `loadClients`**: проверка snake/camel приводит к полному сбросу кэша; нет нормализации на запись, логика трудно сопровождается.
7. **Единичные места без retry/обработки**: часть экшенов опирается на `retryWithExponentialBackoff`, но не все (например, некоторые get/post в меню/инвалидации полагаются на единичный вызов).
8. **Меню и кэш без версионности**: нет явной схемы миграций/версии для localStorage меню; возможен дрейф структуры.
9. **Потенциальные проблемы с async IndexedDB**: `indexedDBStorage` использует async методы, стандартный `vuex-persistedstate` ожидает синхронные; требуется проверка корректности работы.

### 🟢 Мелкие
8. **Неявные зависимости на localStorage при парсинге IndexedDB**: в parser для `projectsData` читается localStorage, ошибки глушатся; потенциально тихий сбой.
9. **Логика фильтров в нескольких местах**: нормализация clientType/cashRegister фильтров разбросана между getter и mutation; можно сконцентрировать.
10. **Смешение ответственного кода**: store содержит бизнес-логику (например, сортировка статусов, фильтрация пользователей по компаниям) — затрудняет тестирование и повторное использование.
